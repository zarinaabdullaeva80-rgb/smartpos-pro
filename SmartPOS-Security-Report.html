🔍 Проблема

До проведения работ система SmartPOS Pro работала как «общий котёл» — все клиенты имели доступ к данным друг друга. Конкретно были обнаружены три критических уязвимости:

1. Ложная привязка в auth.js

В middleware авторизации была строка:

// БЫЛО (опасно):
const organization_id = user.organization_id || 1;
// Если у пользователя не указана организация,
// система считала его частью Организации №1
⚠️ Последствие
ВСЕ новые пользователи автоматически попадали в чужую организацию и видели её данные.

2. Утечка через NULL-записи

Во всех основных маршрутах (товары, продажи, склады) использовался фильтр:

-- БЫЛО (опасно):
WHERE organization_id = $1 OR organization_id IS NULL
-- Записи без привязки были видны ВСЕМ клиентам

3. Глобальная синхронизация 1С

Настройки подключения к 1С (URL, логин, пароль) хранились без привязки к организации. Товары, импортированные одним клиентом из 1С, попадали в общую базу и были видны другим клиентам.

🛠️ Решение: 4 этапа

Этап 1: Фундамент безопасности

Исправление auth.js

// СТАЛО (безопасно):
const organization_id = user.organization_id; // null если не указан
// Теперь без org ID — нет доступа к данным

Строгая фильтрация в 6 маршрутах

-- СТАЛО (безопасно):
WHERE organization_id = $1
-- Каждый клиент видит ТОЛЬКО свои данные

SuperAdmin override

Администратор платформы сохранил доступ ко всем данным для управления:

if (req.user.user_type === 'super_admin') {
    // Видит всё, включая бесхозные записи
} else {
    // Строго только свою организацию
}

Этап 3: Изоляция 1С-синхронизации

Обновлённый сервис sync1cService.js

// БЫЛО:
async getSettings() { ... } // ВСЕ настройки
async processProducts(items) { ... } // Общая куча

// СТАЛО:
async getSettings(organizationId) { ... }
async processProducts(items, organizationId) { ... }

Обновлённая схема базы данных

ALTER TABLE sync_settings ADD COLUMN organization_id INTEGER;
ALTER TABLE sync_log ADD COLUMN organization_id INTEGER;
ALTER TABLE external_id_mapping ADD COLUMN organization_id INTEGER;

-- Уникальные ограничения:
UNIQUE(setting_key, organization_id)
UNIQUE(entity_type, internal_id, external_system, organization_id)
💡 Что это значит
Если у клиента А в 1С товар с ID «ТОВ-001», и у клиента Б тоже есть «ТОВ-001» — они НЕ перепутаются, потому что маппинг привязан к конкретной организации.

Этап 4: Отчёты и дашборд

Тип утечки 1: Некорректный фильтр

-- БЫЛО (опасно):
WHERE p.organization_id = $1 OR p.organization_id IS NOT NULL
-- Показывал товары ВСЕХ организаций!

-- СТАЛО (безопасно):
WHERE p.organization_id = $1
-- Только данные конкретной организации

Тип утечки 2: Условный фильтр

// БЫЛО (опасно):
if (orgId) {
    query += ' AND organization_id = $1'; // Нет orgId = нет фильтра!
}

// СТАЛО (безопасно):
if (!orgId) {
    return res.json({ dashboard: {} }); // Нет org = пустой ответ
}
query += ' AND organization_id = $1'; // Фильтр ВСЕГДА
